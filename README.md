
##  Структура репозитория

-   `init.sql` - скрипт инициализации. Содержит DDL таблиц и DML с тестовыми записями (смоделированы различные сценарии поведения юзеров для проверки логики).
    
-   `solutions.sql` - SQL-запросы с подробными комментариями к каждому шагу.
    

## Логика решений

Разработка велась с упором на модульность и читаемость. Использовался подход с цепочками CTE для прозрачности трансформаций.

-   Задание 1 - Анализ офферов: Решено за один проход через агрегацию с использованием конструкции `FILTER`. Это позволяет избежать громоздких и тяжелых `LEFT JOIN`, оставляя код чистым и производительным.
    
-   Задание 2 - Расчет длины сессий: Реализован классический аналитический паттерн **Gaps and Islands**. С помощью оконных функций `LAG` и `SUM` вычисляются интервалы между событиями. Если лаг превышает заданные 30 минут, генерируется новый `session_id` через нарастающий итог.
    
-   Задание 3 - Когортный анализ по дням: Для гарантированного вывода информации **для каждого дня** прошлой недели применен `generate_series`. Созданный календарь соединяется с агрегированными фактами покупок через `LEFT JOIN`. Это гарантирует, что дни без транзакций не выпадут из отчета, а корректно отобразятся с нулевой долей.
        

## Использование ИИ

В соответствии с условиями задания, для оптимизации времени использовалась ИИ-модель. С её помощью была сгенерирована рутина: бойлерплейт DDL-скриптов и набор DML-инсертов для проверки краевых случаев. 

Архитектурный подход, применение паттернов и финальная отладка SQL-логики выполнены вручную.
